## 一、为什么需要共享状态文件（tasks.json）？

**核心问题**：当用户发送一个复杂任务后，Director 创建了任务 A，但：

1. **任务卡在哪个 Agent 了？** — 如果 Manager 3 分钟没返回，Director 怎么知道卡在哪？
2. **任务结果是什么？** — 汇总时需要获取所有 Worker 的产出
3. **出了问题找谁？** — 审计时需要知道谁做了什么、什么时候做的

**没有状态文件的代价**：即使 subagent 调用是同步返回的，整个协作过程是黑盒，出了问题无法追溯。

**状态文件的简化设计**：
```
tasks.json 每条任务记录的核心字段：
{
  "id": "TASK-xxx",
  "title": "标题",
  "state": "created|doing|done",        // 任务当前在哪
  "owner": "director|manager|worker", // 谁在处理
  "flow_log": [                    // 流转记录（谁→谁）
    {"from": "director", "to": "manager", "at": "时间", "remark": "分配"}
  ],
  "output": "摘要",                // 最终产出
  "createdAt": "时间"
}
```

只有 6 个核心字段，比 edict 的 20+ 字段简化了 10 倍。

## 二、核心设计（两个关键洞察）

### 2.1 来自 edict 的核心机制

edict 的 Agent 协作本质只有两件事：

**① subagent 直接调用（不是 sessions_send）**
```
# 中书省的 SOUL.md 里写的
立即调用门下省 subagent（不是 sessions_send），把方案发过去等审议结果。
立即调用尚书省 subagent，发送最终方案让其派发给六部执行。
```
Agent 之间通过 subagent 机制调用，结果以**文本直接返回**，不需要消息队列、Redis、HTTP 轮询。

**② 共享 CLI 脚本写状态**
```bash
# 所有 Agent 共享同一个脚本，写同一个 JSON 文件
python3 scripts/kanban_update.py state JJC-xxx Doing "正在执行"
python3 scripts/kanban_update.py flow JJC-xxx "中书省" "尚书省" "转派执行"
python3 scripts/kanban_update.py done JJC-xxx "产出" "摘要"
```
各 Agent workspace 通过**符号链接**共享 scripts/ 和 data/ 目录，写同一份 tasks.json。

### 2.2 来自 openclaw-skillspack 的文件结构

每个角色是一个**子技能包**，结构为：
```
<role>/
├── SOUL.md       # 角色身份 + 工作流程 + 命令规范（参考 edict agents/*/SOUL.md）
├── SKILL.md      # 技能描述 + 触发条件（参考 novel-writer/inkos/SKILL.md）
└── _meta.json    # 元数据（参考 novel-writer/inkos/_meta.json）
```

## 三、角色定义（3 个）

```
用户 → Director（决策者）→ Manager（监督者）→ Worker（执行者）
                                  ↓ 结果返回 ↑
```

| 角色 | 对应 edict | SOUL.md 核心规则 |
|------|-----------|-----------------|
| **Director** | 中书省 + 尚书省合并 | 接收任务 → 拆解 → 调用 Manager subagent → 汇总回复 |
| **Manager** | 六部（负责人） | 接收任务包 → 拆解子任务 → 调用 Worker subagent → 汇总返回 |
| **Worker** | 六部（执行者） | 执行具体任务 → 返回结果文本（不调用任何 subagent） |

## 五、agents.json（权限矩阵）

来自 edict 的 `agents.json` 格式，`subagents.allowAgents` 定义谁能调用谁：

```json
[
  {
    "id": "director",
    "name": "director",
    "workspace": "~/.openclaw/workspace-director",
    "subagents": {
      "allowAgents": ["manager"]
    }
  },
  {
    "id": "manager",
    "name": "manager",
    "workspace": "~/.openclaw/workspace-manager",
    "subagents": {
      "allowAgents": ["worker"]
    }
  },
  {
    "id": "worker",
    "name": "worker",
    "workspace": "~/.openclaw/workspace-worker",
    "subagents": {
      "allowAgents": []
    }
  }
]
```

## 六、task_update.py 命令设计

参考 edict `kanban_update.py`，只保留最核心的 4 个命令：

```bash
# 创建任务（Director 收到用户请求时）
python3 scripts/task_update.py create TASK-20260421-001 "任务标题"

# 更新状态 + 流转记录（各 Agent 关键步骤调用）
python3 scripts/task_update.py flow TASK-xxx "Director" "Manager" "任务分配：xxx"

# 实时进展上报（不改状态，只更新当前在做什么）
python3 scripts/task_update.py progress TASK-xxx "正在拆解子任务" "分析🔄|分配|执行|汇总"

# 标记完成
python3 scripts/task_update.py done TASK-xxx "产出摘要"
```

数据写入 `data/tasks.json`，各 Agent workspace 通过符号链接共享同一份文件。

## 七、SOUL.md 核心内容（各角色工作流）

### Director SOUL.md 工作流

```
步骤 1：收到用户请求
  → python3 scripts/task_update.py create TASK-xxx "标题"
  → python3 scripts/task_update.py progress TASK-xxx "正在分析任务" "分析🔄|分配|汇总"

步骤 2：调用 Manager subagent（不是 sessions_send！）
  发送格式：
    📋 Director·任务分配
    任务ID: TASK-xxx
    目标: [一句话]
    子任务:
      - [子任务1描述]
      - [子任务2描述]
    输出要求: [格式/标准]

  → python3 scripts/task_update.py flow TASK-xxx "Director" "Manager" "分配任务"

步骤 3：等待 Manager 返回结果文本

步骤 4：汇总回复用户
  → python3 scripts/task_update.py done TASK-xxx "产出摘要"
```

### Manager SOUL.md 工作流

```
步骤 1：收到 Director 任务包
  → python3 scripts/task_update.py progress TASK-xxx "正在拆解子任务" "拆解🔄|分配|汇总"

步骤 2：调用 Worker subagent
  发送格式：
    📮 Manager·执行指令
    任务ID: TASK-xxx
    具体任务: [描述]
    输出格式: [要求]

  → python3 scripts/task_update.py flow TASK-xxx "Manager" "Worker" "分配执行"

步骤 3：等待 Worker 返回结果文本

步骤 4：汇总返回给 Director（直接返回文本，不用 sessions_send）
  → python3 scripts/task_update.py flow TASK-xxx "Worker" "Manager" "执行完成"
```

### Worker SOUL.md 工作流

```
步骤 1：收到 Manager 执行指令
  → python3 scripts/task_update.py progress TASK-xxx "正在执行任务" "执行🔄|返回结果"

步骤 2：执行任务（调用工具、写文件等）

步骤 3：直接返回结果文本给 Manager
  格式：
    ✅ Worker·执行结果
    任务ID: TASK-xxx
    摘要: [3-5句话]
    产物路径: shared/TASK-xxx/interim/<file>（如有文件）
```

## 九、共享产物目录

```
~/.openclaw/workspace/multi_agents/
├── data/
│   └── tasks.json              # 所有任务状态（共享）
└── shared/
    └── <task_id>/
        ├── interim/            # Worker 中间产物
        └── final/              # Director 最终产物
```

各 Agent workspace 通过符号链接访问：

```bash
# install.sh 中执行
ln -sf ~/.openclaw/workspace/multi_agents/data \
       ~/.openclaw/workspace-director/data
ln -sf ~/.openclaw/workspace/multi_agents/shared \
       ~/.openclaw/workspace-director/shared
# manager / worker 同理
```

## 十、与 edict 的精简对比

| 特性 | edict | 本方案 | 说明 |
|------|-------|--------|------|
| Agent 数量 | 12 个 | **3 个** | 去掉分拣、审核层 |
| 通信机制 | subagent 调用 | **相同** | 核心机制不变 |
| 状态管理 | kanban_update.py（1025行）| **task_update.py（~100行）** | 只保留 create/flow/progress/done |
| 权限矩阵 | agents.json | **相同格式** | 复用 edict 格式 |
| 数据共享 | 符号链接 + data/ | **相同** | 复用 edict 方案 |
| 看板 | React 18 军机处 | **无** | 直接看 tasks.json |
| 消息总线 | Redis Streams | **无** | subagent 直接返回文本 |
| 审核层 | 门下省（专职） | **无** | 精简掉 |

## 十一、实现顺序

1. `scripts/task_update.py` — 核心，所有 Agent 依赖它
2. `agents.json` — Agent 注册 + 权限矩阵
3. `director/SOUL.md` + `director/SKILL.md` + `director/_meta.json`
4. `manager/SOUL.md` + `manager/SKILL.md` + `manager/_meta.json`
5. `worker/SOUL.md` + `worker/SKILL.md` + `worker/_meta.json`
6. `install.sh` — 注册 Agent + 符号链接
