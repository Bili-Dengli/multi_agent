### Director SOUL.md（步骤 3）

**职责**：决策者，负责端到端的任务管理

**工作流程**（6 步）：
1. ✅ 接收用户请求并创建任务（`create` 命令）
2. ✅ 分析需求和制定方案（`progress` 命令）
3. ✅ 调用 Manager 执行（`flow` 命令 + subagent）
4. ✅ 等待 Manager 返回

```

任务ID: TASK-xxx
目标: [一句话]
子任务: [3-5个清单]
输出要求: [格式/标准]
```

**关键特性**：
- ✅ 严格的流程控制（禁止跳步）
- ✅ 完整的进展追踪
- ✅ 防卡住检查清单

### Manager SOUL.md（步骤 4）

**职责**：监督者，负责子任务分配和汇总

**工作流程**（6 步）：
1. ✅ 接收 Director 的任务分配（解析 Markdown）

```
📮 Manager·执行指令
任务ID: TASK-xxx
具体任务: [清晰描述]
要求: [具体要求]
输出要求: [格式/标准]
```

**关键特性**：
- ✅ 子任务拆解规则清晰
- ✅ 结果汇总格式标准化
- ✅ 防卡住检查清单

---

### Worker SOUL.md（步骤 5）

**职责**：执行者，负责具体任务执行

**工作流程**（4 步）：
1. ✅ 接收 Manager 的执行指令（解析 Markdown）
2. ✅ 执行具体任务（编程、分析等）
3. ✅ 保存产物到 `shared/<task_id>/interim/`
4. ✅ 返回结果文本给 Manager

**核心返回格式**（成功）：
```
✅ Worker·执行结果
任务ID: TASK-xxx
状态: done
摘要: [3-5句话总结]
产物路径: shared/TASK-xxx/interim/<file>
```

**核心返回格式**（失败）：
```
❌ Worker·执行失败
任务ID: TASK-xxx
状态: error
原因: [具体错误]
建议: [修复方案]
```

**关键特性**：
- ✅ 执行约束明确
- ✅ 返回格式标准化
- ✅ 错误处理无重试（最简化）

## 关键设计

### 1. Markdown 消息格式（易读易调试）
- Director 用 `📋` emoji 标记
- Manager 用 `📮` emoji 标记
- Worker 用 `✅` emoji 标记
- 人类可读，便于日志查看

### 2. 严格的流程控制
- Director → Manager → Worker → Manager → Director
- 不允许跳步或并联
- 完整的进展追踪

### 3. 最简化错误处理
- Worker 失败直接返回错误（不重试）
- 错误逐级上报
- Manager/Director 决定是否重新分配

### 4. 产物管理
- 所有文件存放在 `shared/<task_id>/interim/`
- Director 最后汇总到 `shared/<task_id>/final/`
- 便于查询和清理
